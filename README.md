# 최재우 - Backend Developer 사전과제 (JVM)

## 기술 스택 및 버전

- Java 17
- Kotlin 1.9.25
- Spring Boot 3.5.11
- Gradle 8
- Spring Security
- Spring Data JPA
- H2 Database
- JJWT 0.11.2
- OpenAI API

## 구현 범위
1. 사용자 인증 
    - 회원가입 API 구현 
    - 로그인 API 구현 
    - JWT 기반 인증 처리 
    - 인증된 사용자 정보를 SecurityContext에서 식별할 수 있도록 구성

2. 대화 생성 
   - 사용자의 질문을 받아 OpenAI API로 전달하는 채팅 생성 API 구현 
   - 대화 요청 시 이전 대화 이력을 함께 전달하도록 구성 
   - 모델명을 요청값으로 받을 수 있도록 구현

3. 스레드 관리 
   - 사용자별 대화 스레드 엔티티 분리 
   - 마지막 대화 시점 기준 30분이 지나면 새 스레드를 생성하고, 그렇지 않으면 기존 스레드에 이어서 대화가 저장되도록 구현 
   - 스레드 목록 조회 API 구현 
   - 스레드 삭제 API 구현

4. 대화 조회 
   - 스레드 단위로 대화를 그룹화하여 조회하는 API 구현 
   - 일반 사용자는 자신의 스레드만 조회 가능하도록 처리 
   - 관리자는 전체 스레드를 조회할 수 있도록 권한 분기 적용 
   - 페이지네이션 및 최신 스레드 기준 정렬 적용

5. SSE 기반 스트리밍 응답
   - OpenAI SSE 응답을 직접 relay 하는 방식으로 SseEmitter 기반 스트리밍 응답 API 구현
        - OpenAI가 전달하는 delta chunk를 실시간으로 클라이언트에 전달하고, 최종 응답은 누적하여 DB에 저장하도록 구성

6. 보안 및 예외 흐름 보완 
   - Spring Security 기반 stateless 인증 구조 구성

### 미구현 기능
- 피드백 생성 / 조회 / 상태 변경 
- 관리자 통계 조회 
- CSV 보고서 생성

3시간이라는 제한된 시간 안에서 모든 요구사항을 동일한 깊이로 구현하기는 어렵다고 판단했다.
대신, 인증 > 대화 생성 > 스레드 관리 > 조회 > SSE 스트리밍까지 이어지는 핵심 사용자 흐름을 안정적으로 완성하는 데 우선순위를 두었다.

## 과제 분석
1. 핵심 사용자 흐름 우선

    3시간이라는 제한된 시간 안에서 모든 요구사항을 동일한 깊이로 구현하기는 어렵다고 판단했다.
    그래서 회원가입, 로그인, JWT 인증, 대화 생성, 스레드 관리, 조회, SSE 스트리밍까지 이어지는 핵심 사용자 흐름을 우선적으로 완성하는 방향으로 범위를 조정했다.


2. 단순 구현보다 확장 가능한 구조를 우선

    과제 설명에서 향후 지속적인 확장 가능성이 중요하다고 판단했다. 그래서 인증, 대화, OpenAI 연동, 조회 기능을 역할별로 분리했고, AI 연동은 서비스 로직 내부에 직접 작성하지 않고 별도 클라이언트 계층으로 분리했다.
    이를 통해 이후 다른 모델이나 외부 AI Provider로 교체하거나, 문서 기반 기능을 추가하더라도 영향 범위를 줄일 수 있도록 했다.


3. 대화와 스레드의 분리

    대화(Chat)와 스레드(Thread)를 하나의 엔티티로 합치지 않고 분리했다.
    그 이유는 이 과제의 핵심 요구사항 중 하나가 30분 기준으로 새 스레드를 생성하는 규칙이기 때문이다.
    이 규칙을 자연스럽게 표현하고, 이후 스레드 단위 조회,삭제,확장 기능을 고려했을 때 스레드를 독립 엔티티로 두는 것이 더 적절하다고 판단했다.


4. 스트리밍 방식으로 SSE를 선택한 이유

    스트리밍 응답은 양방향 통신보다 서버에서 클라이언트로 응답을 순차적으로 전달하는 단방향 통신에 가까웠다. 
   그래서 WebSocket보다 구현 복잡도가 낮고, 현재 요구사항에 더 적합한 SSE 방식을 선택했다.
   또한 OpenAI SSE 응답을 relay 하는 구조를 적용하여, 자연스러운 스트리밍 흐름을 구현했다.


5. Spring MVC + JPA 기반 구조 유지

    Kotlin, Spring Boot, JPA 기반 과제에서 전체 구조를 Reactive 스택으로 확장하기보다, 익숙하고 안정적으로 구현 가능한 Spring MVC + JPA + SSE 조합을 유지했다.
    이를 통해 인증, DB 저장, SSE 응답 흐름을 하나의 일관된 구조 안에서 구현하는데 목표를 두었다.

## 과제 수행에서 AI의 활용
이번 과제에서는 AI를 단순한 코드 생성 도구가 아니라, 제한된 시간 안에서 구현 속도를 높이기 위한 개발 도구로 적극 활용했다.

과제는 3시간 안에 요구사항 분석, 구현 우선순위 결정, 실제 기능 개발, 결과 보고서 작성까지 모두 마쳐야 했다. 이 시간 제약을 고려하면, 반복적인 보일러플레이트나 문법 전환 작업에 시간을 많이 쓰기보다, AI를 활용해 구현 속도를 높이고 나는 더 중요한 설계 판단과 검증에 집중하는 것이 더 합리적이라고 판단했다.

특히 Java에는 익숙하지만 Kotlin에는 상대적으로 익숙하지 않았기 때문에, 다음과 같은 영역에서 AI를 적극적으로 활용했다.

1. Java 스타일로 먼저 떠오른 구현 아이디어를 Kotlin 코드 형태로 빠르게 변환하는 작업

2. DTO, 설정 클래스, JWT 관련 보일러플레이트, 컨트롤러/서비스 기본 골격 작성

3. Spring Security 설정, SSE 관련 기본 코드 구조, OpenAI 연동용 초안 코드 작성

4. 에러 로그를 바탕으로 원인 후보를 빠르게 좁히고 수정 방향을 탐색하는 작업


다만 이번 과제에서 중요한 것은 단순히 코드를 빠르게 만들어내는 것이 아니라, 무엇을 먼저 구현할지, 어떤 구조로 나눌지, 어떤 기술을 선택할지 결정하는 것이라고 생각했다. 따라서 아래의 내용은 직접 판단하였다.

1. 아키텍처 구성
    
    이번 과제에서는 WebFlux나 Reactive 스택까지 확장하기보다, Spring MVC + JPA + SSE 조합으로 가는 것이 과제 범위와 시간 제약에 더 적절하다고 판단했다. JPA를 사용하는 이상 전체 구조를 무리하게 reactive로 가져가는 것은 일관성이 떨어질 수 있고, 과제의 핵심은 안정적인 완성이라고 생각하였다.


2. 실행 흐름에 대한 결정

    채팅 생성 기능에서는 단순히 질문과 응답만 저장하는 것이 아닌, 이전 대화 이력을 함께 전달하고 마지막 대화 시점 기준 30분이 지나면 새 스레드를 만드는 흐름을 직접 설계했다. 이는 과제 요구사항을 그대로 옮기는 것처럼 보일 수 있지만, 실제로는 엔티티 분리, 조회 방식, 저장 시점, 이후 조회 API 구조까지 함께 고려해야 하는 문제였다. 이 흐름은 AI가 임의로 만든 구조를 그대로 사용한 것이 아니라, 요구사항을 읽고 내가 먼저 흐름을 정리한 뒤 그에 맞게 구현을 구체화하였다.


3. 기술 선택의 방향

   스트리밍 기능에서도 처음에는 완성된 응답을 분할해서 SSE로 전송하는 방식으로 시작했지만, 그 방식은 진짜 스트리밍이 아니라 판단했다. 그래서 이후에는 OpenAI의 SSE 응답을 직접 받아 relay 하는 구조로 변경했다. 이 과정에서 무엇이 더 구현 난이도가 높고, 무엇이 과제 요구사항에 더 가까운지 비교했으며, 최종적으로는 실제 SSE relay 방식이 더 적절하다고 판단해 구조를 수정했다.


4. AI를 사용할 때도 결과를 그대로 받아들이지 않고, 항상 검증하고 수정함

   Kotlin 코드나 Security 설정 코드는 빠르게 얻을 수 있었지만, 프로젝트의 Gradle 버전, 라이브러리 버전 차이, SSE 이후의 예외 흐름처럼 초안 그대로는 맞지 않는 부분이 나왔다. 이런 부분은 로그를 직접 확인한 뒤, 현재 프로젝트 상황에 맞게 코드를 다시 수정하면서 적용했다.


## 가장 어려웠던 기능/구현

이번 과제에서 가장 어려웠던 기능은 SSE 기반 스트리밍 응답 구현이었다.

처음에는 단순히 OpenAI의 일반 응답을 모두 받은 뒤, 이를 일정한 길이로 잘라 SseEmitter를 통해 순차적으로 전달하는 방식으로 접근했다. 이 방법은 빠르게 동작을 확인하기에는 적합했지만, 실제 스트리밍이라고 보기에는 한계가 있다고 판단했다. 그래서 이후에는 OpenAI의 SSE 응답을 서버가 직접 받아서, 그 데이터를 다시 클라이언트로 relay 하는 방식으로 구조를 변경했다.

이 과정에서 가장 중요했던 부분은 두 가지였다.

첫 번째는 OpenAI의 스트리밍 응답 조각을 실시간으로 처리하면서도, 최종 답변 전체를 누적해 저장해야 한다는 점이었다. 스트리밍 응답은 조각 단위로 도착하기 때문에 화면에는 바로 전달할 수 있지만, DB에는 최종적으로 완성된 답변 하나를 저장해야 했다. 그래서 스트리밍 중에는 chunk를 즉시 전송하면서도, 동시에 StringBuilder로 전체 응답을 누적하는 방식으로 처리했다.

두 번째는 Spring Security와 비동기 SSE 처리의 결합에서 발생한 예외 흐름이었다. 실제로 스트리밍 응답은 정상적으로 끝났고 done 이벤트까지 클라이언트에 전달되었지만, 이후 서버 로그에서 response is already committed, Access Denied 예외가 발생했다. 원인을 확인해보니 SSE의 async dispatch 과정에서 /error로 흐름이 넘어가면서 Security 예외가 다시 발생하는 문제였다. 이를 해결하기 위해 /error 경로를 허용하고, JWT 필터가 async dispatch 상황에서도 동작하도록 보완했다.
